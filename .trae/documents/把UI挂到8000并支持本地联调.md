# 现象解释（为什么你没看到UI）

你截图里看到的页面其实是**后端 API 的首页**（我给 FastAPI 增加了一个简单的 HTML landing page），所以它只会显示：
- `/docs`、`/health` 以及 `/rewrite` 的示例

真正的 UI 在前端工程 `web/` 里，它默认由 Vite 启动在 **5173 端口**：
- 配置位置：[vite.config.ts](file:///d:/12/web/vite.config.ts#L29-L33)
- UI 入口组件：[App.tsx](file:///d:/12/web/src/App.tsx)

你访问 `http://127.0.0.1:8000/` 地址栏变成 `http://localhost:8000/`，一般是浏览器/扩展/代理把 127 映射成 localhost（两者指向同一台机器，核心问题不是重定向，而是你访问的是 API 端口）。

# 你现在立刻能看到UI的方式（不执行，只给操作指引）

1) 先启动后端（8000）：
- 在 `d:\12\backend` 运行 `uvicorn app.main:app --host 127.0.0.1 --port 8000`

2) 再启动前端（5173）：
- 在 `d:\12\web` 运行 `npm i` 然后 `npm run dev`

3) 打开 UI：
- 访问 `http://localhost:5173/`

# 具体开发改造计划（你确认后我再执行）

## 1. 本地联调体验
- 在 `web` 侧新增开发代理：`/api/*` 反代到 `http://127.0.0.1:8000`，这样前端请求同源，不需要依赖 CORS 或环境变量。
- 前端把请求地址从 `${apiBase()}/rewrite` 改成固定走 `/api/rewrite`。

## 2. 后端一口端口提供 UI + API
- 后端增加静态资源托管：把 `web` 的 `dist/` 作为静态目录挂到 `/`。
- 增加 SPA fallback：任意未知路径返回 `index.html`（保证刷新不 404）。
- API 路由统一前缀 `/api`，避免和前端路由冲突。

## 3. PWA 正式可用
- 确保 build 后的 `manifest`、service worker 正常产出；并在后端静态托管时正确带上缓存头。
- 在 README 里说明：PWA 在生产环境建议 HTTPS（localhost 例外）。

## 4. 验证与回归
- 增加一个端到端校验脚本：同时检查 `/` 返回 UI HTML、`/api/health` 200、`/api/rewrite` 200。
