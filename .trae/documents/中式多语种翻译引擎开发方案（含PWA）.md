# 中式多语种翻译引擎开发方案（含PWA）

## 现有仓库盘点（基于当前文件夹内容）
- 现状定位：当前目录以“数据与整理脚本”为主，还不是可运行的“引擎/应用”。
- 已有资产：
  - 语料/词典：THUOCL 中文词库、WordNet 英文语义词典、Tatoeba 中英平行语料（含 links/小样本）。
  - 规则/案例：中式英语对照表（Markdown 表格，偏“搭配/句式”案例库）。
  - 工程化思路文档：提出“标准语言 → 语义IR → 分层规则/模板 → 后处理”的整体框架与 6 层生成体系。
  - 脚本：主要做清洗/去重/重写 Markdown 文件，并未形成可扩展的规则引擎或 API。

## 产品定义（你要做的到底是什么）
- 目标：把“中文思维的表达方式”投射到目标语言中，生成“中式英语/中式法语/中式西语…”等风格化文本，同时保持可读、可控、可复现。
- 核心能力：
  - 多语言输入/输出：支持主流语言（优先英/法/西/德/日/韩等可逐步扩展）。
  - 风格强度：L1-L5（轻度→重度中式），并支持领域权重（技术/旅游/政务/客服等）。
  - 可解释：输出“应用了哪些规则/模板、概率、冲突解决结果”。
  - 可验证：有自动评测与回归集，规则变更可回放对比。
- 三种使用模式（建议都支持，先做其中一个 MVP）：
  - A. 同语改写：标准法语→中式法语（不改变语言，只改表达习惯）。
  - B. 直译生成：中文→中式英语/中式法语（更贴近“中文表达方式”）。
  - C. 任意语言→中式目标语言：先做语义归一/翻译到“标准目标语言”，再做中式风格变换。

## 总体架构（可落地、可扩展）
- 推荐采用“插件化语言包 + 分层规则引擎”的架构：
  - 前端：PWA（离线可用、可安装），用于交互、管理规则包、展示解释与对比。
  - 后端：API 服务（规则执行、语义IR、可选的翻译/模型推理、评测服务）。
  - 资源层：规则包（JSON/YAML）、词典与映射表（本地文件/SQLite/内存索引）、语料与评测集。

## 核心引擎设计（从“文档思路”变成“可跑代码”）
- 处理流水线（与现有文档一致，但工程化）：
  1) 语言检测：识别输入语言、脚本与混合语言。
  2) 语义标准化（Normalization）：把复杂句、习语、短语动词等降到可控形式（为规则/模板稳定性服务）。
  3) 语义中间表示 IR：
     - 最小可行 IR：句子切分 + 依存结构 + 关键信息槽（主谓宾、时间地点方式、因果转折、否定、疑问）。
     - 扩展 IR：语义角色（SRL）、指代消解、篇章结构（背景→结论）。
  4) 6 层生成（建议严格分层、可配置开关与概率）：
     - L0 中文思维模板（背景先行、过程强调、话题-评论结构）
     - L1 句法结构映射（状语前置、主语省略/泛化等）
     - L2 词汇与搭配直译（基于现有中式对照表 + 规则扩展）
     - L3 形态弱化/破坏（时态/冠词/介词/一致性简化）
     - L4 修辞与文化迁移（敬语直译、修辞直译、语气词映射等）
     - L5 组合与强度控制（规则冲突、顺序、随机种子、领域权重）
  5) 后处理过滤：可读性阈值、禁用极端错误、重复词/标点修复。
- 规则引擎落地方式：
  - 规则 Schema（JSON）+ 运行时执行器：trigger（条件）+ transform（操作）+ probability（强度/领域）+ conflicts（冲突图）。
  - Transform 采用“原子操作库”：重排、替换、插入冗余、删除虚词、形态简化等。
  - 冲突解决：优先级 + 互斥组 + 最小可读性评分。

## 多语言扩展策略（“中式英语”到“中式多语种”）
- 语言包（Language Pack）标准接口（每种语言实现一套）：
  - tokenizer / sentence splitter
  - POS/依存分析（或轻量规则解析器）
  - morphology（变形/还原/一致性）
  - generation helpers（该语言常见固定结构与易错点）
- 优先级路线：
  - 第 1 个语言：英语（你已有对照表 + WordNet 可直接做词汇层）。
  - 第 2-3 个语言：法语/西语（形态较多，重点做 L3“弱化/错配”的可控版本）。
  - 后续：德语/俄语/阿语等（需要更强形态处理，建议先做“轻度中式”保证可读）。

## 数据与知识库建设（复用现有 datasets/）
- 把现有 Markdown 对照表升级为“可机器消费”的词汇/搭配库：
  - 统一字段：source_phrase、target_phrase、category、layer、domain、severity、examples。
- 用现有 Tatoeba 平行语料做两类事情：
  - 回归集：固定输入→固定输出（给每个规则/组合建立测试样例）。
  - 评测集：抽样句子做自动指标与人工验收。
- 词典利用：
  - WordNet：同义替换候选、词形还原、避免替换破坏语义。
  - THUOCL：中文侧领域词识别（用于“中文→中式外语”的领域权重与术语策略）。

## 评测与质量门禁（保证“正确”与可迭代）
- 三类指标：
  - 可理解度：目标语言语言模型困惑度/句法完整度、人工可读评分。
  - 中式程度：规则覆盖率、中文语序特征触发率、冗余/直译特征计数。
  - 语义保持：关键词/实体保持率、否定/时序/因果不被破坏的检查。
- 回归机制：规则/词库每次变更都跑同一批用例，输出差异报告。

## API 与前端（PWA）设计
- API（建议先定 1-2 个核心接口）：
  - POST /rewrite：{text, lang, level, domain, seed, explain=true} → {output, applied_rules, warnings}
  - POST /translate：{text, source_lang, target_lang, level, domain, seed, explain} → 同上
  - GET /rulepacks：列出规则包与版本
  - POST /rulepacks/validate：规则包校验与冲突检测
- PWA 关键页面：
  - 输入/输出对照（支持“标准→中式”双栏、复制、导出）。
  - 强度/领域/随机种子控制（可复现）。
  - 解释面板（规则链路、每步变换前后 diff）。
  - 历史记录（本地存储，离线可查）。
- PWA 技术要点：
  - Service Worker 缓存静态资源与最近 N 次请求结果（离线仍可浏览历史/规则说明）。
  - manifest + 可安装 + 断网提示与降级策略（离线时仅提供“本地轻量规则包”或历史回放）。

## 工程落地与目录规划（把当前仓库升级成项目）
- 建议新增顶层模块（不影响现有 datasets/）：
  - backend/：引擎与 API 服务
  - web/：PWA 前端
  - rulepacks/：规则包与模板库（按语言/版本组织）
  - eval/：评测集、指标脚本、回归报告
- 现有脚本迁移：将清洗脚本归档到 tools/，并把产物改为生成 rulepacks 的数据源。

## 里程碑（按可交付切分，先小后大）
- MVP-1（最小可用）：英语同语改写（标准英语→中式英语），支持 L1-L5、解释输出、PWA 前端。
- MVP-2：中文→中式英语（更符合“采用中文表达方式”），并引入领域权重。
- V1：法语/西语语言包 + 多语言统一接口 + 评测与回归体系。
- V2：引入“模型辅助”（可选）用于 IR 或候选生成，但保持规则层可控。

## 风险与对策（提前保证可控性）
- 多语言形态复杂导致输出崩坏：先限制为轻度中式（L1-L3）并启用后处理阈值。
- 规则组合爆炸：强制分层 + 冲突图 + 规则互斥组 + 种子可复现。
- 数据不足：先用“人工规则 + 小规模高质量对照库”起步，再逐步半自动归纳。
